# Database Best Practises

> **Audience:** every agent (and every human) writing code, migrations, or queries against the rebuild's database. This is the canonical "do / don't" list for the rebuild project.
> **Authoritative parents:** [`rebuild.md`](../../rebuild.md), [`rebuild/plans/m0-foundations.md`](m0-foundations.md), [`rebuild/plans/MYSQL_FEATURE_AUDIT.md`](MYSQL_FEATURE_AUDIT.md). Where this file conflicts with those, those win and this file should be patched.

## A. General database engineering â€” universal dos and don'ts

These apply to **any** SQL database; if you're tempted to break one of them, escalate in PR rather than silently doing it.

### A.1 Schema design

**DO**

- **Normalise first, denormalise only with evidence.** Start with one fact in one row in one table. Denormalise (a redundant column, a JSON blob, a materialised aggregate) only after a real query plan or `EXPLAIN ANALYZE` shows a measured cost.
- **Make every relationship explicit with a FOREIGN KEY constraint.** Even when "the application enforces it" â€” the FK is your last line of defence against orphans, your rollback insurance, and your documentation.
- **Pick `ON DELETE CASCADE` vs `ON DELETE SET NULL` deliberately.** "Folder is deleted â†’ its child folders die" = `CASCADE`. "Folder is deleted â†’ its chats survive but lose the folder pointer" = `SET NULL`. Write the choice into the migration's docstring so the next agent knows it was a choice, not an accident.
- **Constrain enumerations.** `CHECK (status IN ('pending','running','success','error'))` (or a typed `ENUM`) is cheap insurance against a typo three milestones from now.
- **Pin a single timestamp convention project-wide.** This project uses `BIGINT` epoch milliseconds (UTC) generated by `app.core.time.now_ms()` (see `rebuild.md` Â§4). Never mix that with `DATETIME`, `TIMESTAMP`, or seconds-precision integers in the same project.
- **Use UUIDv7 for all identifiers.** Generated app-side via `app.core.ids.new_id()` and stored as `VARCHAR(36)` â€” i.e. `mapped_column(String(36), ...)` in SQLAlchemy. **Never `CHAR(36)`** for UUIDs: the project-wide locked decision in `rebuild.md` Â§9 is `VARCHAR(36)`, and mixing the two across tables can fail FK creation in MySQL 8.0 InnoDB (the engine checks referenced and referencing column types match). `CHAR(N)` is reserved for genuinely fixed-width *non*-UUID values like SHA-256 hex digests (`CHAR(64)`). Don't reach for autoincrement `INTEGER` PKs (no global uniqueness, painful cross-shard, painful merging) or `uuid4` (no insertion locality).
- **Constrain at the column where you can.** `NOT NULL`, `UNIQUE`, `CHECK` all run inside the storage engine and cost essentially nothing per row. The application layer enforces them too â€” both belt and braces.
- **Name everything.** Every constraint, every index. `ix_chat_user_updated`, `uq_user_email`, `fk_chat_share_id`, `ck_channel_message_one_author` â€” the name appears in slow-query logs, lock-wait views, and `pg_dump` output. Anonymous constraints are a debugging tax.
- **Write the migration so a human can read it.** Every revision file gets a docstring explaining *why* (not just what). The diff to your model file does not survive `git log --follow` half as well.

**DON'T**

- **Don't use `SELECT *` in application code.** It locks you into the current column set, ships bytes you don't need (especially for blob/text columns), and silently breaks when columns are added or reordered. Spell the columns out â€” it's three more characters and a contract with the schema.
- **Don't store derived values without a generated column or trigger.** Two columns that are supposed to agree, but where one is computed in Python on insert, *will* drift. Use a `STORED` generated column or accept the cost of computing on read.
- **Don't bury foreign-key cascades you'd be surprised by.** `ON DELETE CASCADE` is correct for "owned" relationships and dangerous for "referenced" ones. Audit cascades when you add new tables that point at existing ones.
- **Don't reuse a column meaning** ("`status='3'` used to mean 'archived', now it means 'pending review'"). New meaning â†’ new column or new enum value, never silent reinterpretation.
- **Don't put unbounded text in an indexable column.** `TEXT` columns in indexes need a prefix length, lose ordering, and are a frequent foot-gun. If you need to index it, capture a deterministic projection in a generated column.

### A.2 Indexing

**DO**

- **Compose indexes column-order-by-selectivity-then-sort-order.** `(user_id, updated_at DESC)` for "list this user's chats by recency" â€” `user_id` first because it filters most rows; `updated_at` last because it satisfies the `ORDER BY`.
- **Cover the predicate AND the projection where it matters.** A query that selects 5 columns and filters on 2 will still hit the table heap unless those 5 columns are in the index. For a hot path, that extra IO is the whole cost.
- **Match the index to the actual workload.** Add an index when a real query plan calls for it; remove (or hide) it when it stops paying. Indexes cost write throughput on every `INSERT`/`UPDATE` and storage on every secondary index (which carries the PK).
- **Plan for the slow query you don't have yet.** Know which queries are critical and which are best-effort. Sidebar list = critical, full-text search = best-effort.
- **Use partial / functional indexes for "hot subset" queries.** If 99% of `chat` rows have `archived = FALSE`, an index on `(user_id, updated_at) WHERE archived = FALSE` (Postgres) or `(user_id, archived, updated_at)` (MySQL) is dramatically smaller and faster than the full index.
- **Drop unused indexes â€” but make them invisible first.** `INVISIBLE` lets the optimiser ignore the index for a week of real traffic before you commit to dropping it. (See B.6.)

**DON'T**

- **Don't index every column "just in case".** Each index is overhead on every write and storage on the secondary tree. The right number of indexes is "the smallest set that makes the critical-path queries fast".
- **Don't index huge `BLOB`/`TEXT` columns.** If you need to look up by a hash or prefix, store the hash/prefix as a separate generated column and index that.
- **Don't trust the optimiser blindly.** Run `EXPLAIN ANALYZE` (Section B.5) on every new hot-path query. Estimates and reality often diverge.
- **Don't add an index in the same revision that drops a different index.** Land the add, observe a week, then land the drop. Atomicity is for the *application*, not for *learning*.

### A.3 Query patterns

**DO**

- **Use parameterised queries always.** SQL injection is solved by the driver if you use bind parameters; it is *not* solved by string interpolation no matter how careful you are.
- **`LIMIT` every query that could ever return more than a screenful.** A missing `LIMIT` on `GET /api/chats` is a latent OOM at scale-up time.
- **Paginate via cursor on `(sort_column, id)`, not `OFFSET N`.** `OFFSET` re-scans every skipped row; cursors don't.
- **Wrap multi-statement work in a transaction.** "Insert run, update automation" is one logical unit; split commits leave half-states on a crash.
- **Hold transactions short.** Long-held row locks are the source of every "why is this query suddenly stalling" mystery. Keep `BEGIN ... COMMIT` to the smallest scope that preserves the invariant.
- **Use atomic upsert when racing for the same row.** `INSERT ... ON DUPLICATE KEY UPDATE` (MySQL) / `INSERT ... ON CONFLICT DO UPDATE` (Postgres) is race-free; the read-then-write pattern isn't.

**DON'T**

- **Don't `SELECT ... FOR UPDATE` without `SKIP LOCKED` (or `NOWAIT`) for queue-style work.** Plain `FOR UPDATE` makes workers wait on each other. (See B.3.)
- **Don't issue N+1 queries from application code.** "Get chat, then per chat get user, then per user getâ€¦" is the single most common performance bug. Use a `JOIN`, a CTE, or a single `IN (...)` lookup.
- **Don't cast or wrap a column in a function in a `WHERE` clause unless you have a functional index for it.** `WHERE LOWER(email) = ?` defeats `ix_email`. Use a functional index or compute the projection at write time.
- **Don't hold a transaction open across a network call** (HTTP, RPC, message bus, model gateway). The other side is allowed to take 30 seconds; your row locks are not.
- **Don't write data in a `SELECT`-driven loop.** Batch the writes (`INSERT ... VALUES (?), (?), (?), ...`); use one transaction; if the dataset is huge, chunk in fixed-size batches with explicit commits.

### A.4 Migrations

**DO**

- **Make every migration idempotent and re-runnable.** A retry on a half-applied migration must be a no-op for the parts that already landed and complete the rest. The rebuild enforces this via the M0 helpers (`*_if_not_exists` / `*_if_exists`); see [`m0-foundations.md` Â§ Migration helpers](m0-foundations.md).
- **Make every migration reversible.** Every `upgrade()` has a `downgrade()`. Even if you never plan to run it, the existence of the function forces you to think about whether you can.
- **Land schema and data changes in separate revisions.** "Add column" then "backfill it" then "tighten constraints" â€” three revisions, three rollback points, three independent retries. Big-bang revisions are a foot-gun.
- **Prefer additive over breaking.** Add the new column nullable, dual-write from app, backfill, switch reads, drop the old column. Each step is reversible; the all-at-once isn't.
- **Test the migration against production-shape data before deploy.** A dev DB with 100 rows hides every interesting failure mode.
- **Pin the DDL algorithm.** On MySQL, `ALGORITHM=INSTANT, LOCK=DEFAULT` is the right default for `ADD COLUMN`; on Postgres, prefer `ALTER TABLE ... ADD COLUMN ... DEFAULT NULL` (no rewrite) over `ADD COLUMN ... DEFAULT 'x' NOT NULL` (full rewrite). Knowing which DDL rewrites the table is on you, not on the migration tool.

**DON'T**

- **Don't put data backfills inside the same revision as the DDL change.** A failed backfill on a half-built schema is the worst recovery scenario in the book.
- **Don't run DDL inside a transaction expecting rollback to undo it.** MySQL DDL implicitly commits the surrounding transaction; Postgres DDL is mostly transactional but every dialect has exceptions. Read the manual for the specific operation.
- **Don't `ALTER TABLE` a hot table during peak hours** without knowing the algorithm + lock combination. The default falls back to `COPY`, which holds an exclusive lock for the duration of a full table copy.
- **Don't drop a column the application might still read.** Deploys are not atomic across pods. Drop *after* every running version has stopped referencing the column.

### A.5 Performance and observability

**DO**

- **Measure before optimising, measure after.** "I think this is faster" is not a basis for shipping. `EXPLAIN ANALYZE`, the slow-query log, and a Grafana panel are the basis.
- **Log slow queries.** A 1-second slow-query threshold in dev catches bugs early; production thresholds depend on the workload but should never be off.
- **Pool connections, but bound the pool.** A pool of 200 to a database that can handle 30 active connections is a guarantee of incident.
- **Set per-statement timeouts.** A query that runs for 5 minutes is almost always a bug; let the database kill it. (See B.5 for `MAX_EXECUTION_TIME`.)
- **Watch index bloat, table bloat, and replication lag** as routine SLIs, not as incident discovery.

**DON'T**

- **Don't run `OPTIMIZE TABLE` / `VACUUM FULL` / `REINDEX` casually.** They lock or rewrite. There's a reason your DB doesn't do them automatically.
- **Don't leave debug `EXPLAIN`/`pg_stat_statements`-style instrumentation enabled in tight loops.** Performance schema collection costs cycles; use it when needed.
- **Don't tune one knob at a time without baseline.** "I bumped `innodb_buffer_pool_size`" with no before/after numbers tells you nothing.

### A.6 Operational hygiene

**DO**

- **Back up. Test the restore.** A backup you've never restored is not a backup.
- **Keep replicas in sync with monitoring on lag.** If you read from a replica, you must alert when its lag exceeds the staleness budget.
- **Version-control configuration.** Every server tuning lives in source control (Helm values, MySQL config map, `mysqld-auto.cnf` mirror â€” see B.7).
- **Run schema migrations as a separate, gated job.** Not as an init container, not at app startup. Failure â†’ deploy fails, app keeps serving the old version.
- **Keep secrets out of the SQL.** No DB-side encryption keys, no plaintext tokens, no auth secrets in seed data.

**DON'T**

- **Don't ALTER production from a psql/mysql REPL.** Always go through a versioned migration.
- **Don't grant `SUPER`/`ALL PRIVILEGES` to the app user.** App users get the minimum: `SELECT, INSERT, UPDATE, DELETE` on their schema, plus `EXECUTE` on whatever stored routines exist. DDL belongs to a separate migration role.
- **Don't ship credentials in the image, the repo, or the env file checked into git.** Sealed secrets, vault, k8s `Secret` â€” pick one and use it.

### A.7 Security

**DO**

- **Always parameterise.** (Restated because it's the single highest-leverage rule.)
- **Validate input shapes (Pydantic / equivalent) before they hit the DB.** Length limits, type constraints, regex where appropriate.
- **Rate-limit at the application layer, not the DB.** Per-user quotas in Redis / sliding-window â€” not via DB triggers.
- **Hash secrets at rest.** Webhook tokens and share tokens are stored as SHA-256 (M3); plaintext never persists. Same rule for any future credential.
- **Redact logs.** Tokens, emails, anything PII gets a redaction filter before it leaves the process.

**DON'T**

- **Don't trust the client about identity.** The trusted-header pattern (`X-Forwarded-Email`) is gated by a trusted-IP allowlist (see [`m5-hardening.md` Â§ Trusted-header verification](m5-hardening.md)). Outside that allowlist the header is stripped before the auth dependency runs.
- **Don't echo user-supplied data back into a query without escaping.** `LIKE '%' || userInput || '%'` is fine via parameter binding; concatenated into SQL it isn't.
- **Don't enable user-controllable SQL fragments.** No "advanced search" backdoor that takes a raw `WHERE` clause from the UI.

## B. MySQL 8.0 specifics â€” what to use, how, and what to avoid

The rebuild targets **MySQL 8.0** specifically (8.0.39 in compose; whatever the managed flavour pins in prod). Everything below assumes that version.

### B.1 Storage engine, charset, collation, packet size

**DO**

- **InnoDB only.** Every table. Every index. (`mysql_engine="InnoDB"` is set as the default in `create_table_if_not_exists`.)
- **`utf8mb4` / `utf8mb4_0900_ai_ci`** as the database default. Never `utf8mb3` (the legacy 3-byte-only encoding); it can't store emoji or many CJK glyphs and is deprecated.
- **`max_allowed_packet=16M`** server-side, with the application enforcing the file cap at 5 MiB so we never come close to the limit. (See `rebuild.md` Â§9.)
- **`innodb_strict_mode=1`, `sql_mode=STRICT_ALL_TABLES,NO_ENGINE_SUBSTITUTION`.** Strict mode rejects truncating writes and silent type coercion â€” exactly what you want.
- **`local_infile=0`.** No client-side `LOAD DATA LOCAL INFILE`; it's a path-traversal foot-gun.

**DON'T**

- **Don't use `MyISAM`, `MEMORY`, `ARCHIVE`, or `CSV` storage engines.** They lack transactions, FK enforcement, and crash-safety. The few use cases they had are better served by InnoDB plus a temp table or a Redis sorted set.
- **Don't pin `latin1` or `utf8mb3` collations on individual tables.** Inherit from the database default; if you ever need to override, write a comment explaining why.

### B.2 Identifiers, JSON, generated columns

**DO**

- **UUIDv7 strings as `VARCHAR(36)` for every PK.** Generated by `app.core.ids.new_id()`. The leading 48-bit ms timestamp gives clustered B-tree inserts; the string form is debug-friendly.
- **Native `JSON` columns when the shape is genuinely free-form.** `chat.history` and `channel_message.content` are good fits â€” they're trees / variant content with no useful relational shape. Don't shoehorn relational data into JSON because it's "easier".
- **`STORED` generated columns + ordinary indexes for hot JSON-path lookups.** The pattern is in `chat.current_message_id`:
  ```sql
  ADD COLUMN current_message_id VARCHAR(36)
  GENERATED ALWAYS AS (JSON_UNQUOTE(JSON_EXTRACT(history, '$.currentId'))) STORED;
  CREATE INDEX ix_chat_current_message ON chat (current_message_id);
  ```
  Don't try to put a functional index directly on `JSON_EXTRACT(...)` â€” the generated column form is clearer and the optimiser handles it identically.
- **`CHECK` constraints for invariants the application enforces.** `((user_id IS NOT NULL) + (bot_id IS NOT NULL) + (webhook_id IS NOT NULL)) = 1` on `channel_message` is the canonical example. MySQL 8.0.16+ actually enforces them (8.0.0â€“8.0.15 silently ignored).

**DON'T**

- **Don't reach for `BINARY(16)` UUIDs (`UUID_TO_BIN(?, 1)`).** Considered and rejected for this project (`rebuild.md` Â§9 + `MYSQL_FEATURE_AUDIT.md` Â§2). UUIDv7 strings already give the locality benefit; the binary form costs `TypeDecorator` complexity that doesn't pay back at internal scale. Revisit only if benchmarks demand it.
- **Don't call `uuid.uuid4()` directly in application code.** Use `app.core.ids.new_id()`. Ruff's `flake8-tidy-imports` bans the direct call under `app/`; tests are exempt.
- **Don't store relational data as JSON to "save a join".** A `channel_message.reactions = ['+1', 'heart']` JSON array seems convenient until you need "who reacted with `heart` on this channel today" â€” then it's a full table scan. Reactions live in `channel_message_reaction` (a row table) for exactly this reason.
- **Don't put generated columns in `__table_args__` and hope SQLAlchemy emits the right DDL.** SQLAlchemy 2 has no first-class `GENERATED ALWAYS AS (...)`. Emit the DDL via `op.execute` (wrapped in `execute_if`) per the M0 migration helpers.

### B.3 Concurrency: locking, queues, atomicity

**DO**

- **`SELECT ... FOR UPDATE SKIP LOCKED` for every queue-style claim.** The APScheduler tick (`m4-automations.md`) is the canonical example. Multiple workers polling the same `automation` rows each get a disjoint subset; nobody waits.
- **`INSERT ... ON DUPLICATE KEY UPDATE` for upsert.** First-login user creation in `get_user` (M0) uses this against the `email` unique index. Race-safe at the index level.
- **Short transactions, scoped to one logical unit of work.** Open `BEGIN`, do the work, `COMMIT` â€” within milliseconds, not seconds.
- **`MAX_EXECUTION_TIME(<ms>)` optimizer hint on any query that must not stall.** The scheduler tick is bounded to 2000 ms (`m5-hardening.md`). Same hint can wrap any read-only path that has a hard SLO.

**DON'T**

- **Don't `SELECT ... FOR UPDATE` (without `SKIP LOCKED`) on a shared queue.** All workers serialise on the first contended row.
- **Don't open a transaction, call an external API, then commit.** The external call can take 30s; your row lock blocks every other worker for those 30s.
- **Don't rely on `LOCK TABLES`.** It's a sledgehammer; row locks via `FOR UPDATE` are surgical and the right tool for almost every case.
- **Don't pass user input into `MAX_EXECUTION_TIME` or any other optimizer hint.** Hints are SQL injection vectors if templated naively.

### B.4 Migrations on MySQL specifically

**DO**

- **Use the M0 `*_if_not_exists` / `*_if_exists` helpers, never bare `op.*`.** Enforced by AST grep gate (`tests/test_migrations.py::test_no_bare_op_calls`). See [`m0-foundations.md` Â§ Migration helpers](m0-foundations.md).
- **`ALGORITHM=INSTANT, LOCK=DEFAULT` for `ADD COLUMN`** unless the column type forces INPLACE/COPY. INSTANT is metadata-only; runtime independent of table size. The helper `add_column_if_not_exists` defaults to INSTANT and fails-fast if MySQL can't honour it (rather than silently downgrading to a multi-hour COPY).
- **Pin `mysql_engine`, `mysql_charset`, `mysql_collate` table args** (the helper does this for you). Tables that drift to default charset are a future migration headache.
- **Use a separate Alembic revision for each schema concern.** "Add column", "backfill", "tighten constraint", "drop old column" â€” four revisions, four retry points.
- **Test that `alembic upgrade head` is idempotent on a fully-upgraded DB and that `alembic downgrade base` is symmetric.** Both gates ship in M0 (`test_upgrade_head_is_idempotent`, `test_downgrade_base_is_idempotent`).

**DON'T**

- **Don't put DDL inside a Python `with session.begin():` and expect rollback.** MySQL DDL implicitly commits. Use Alembic's revision-level transaction or accept that DDL is non-transactional and design the migration to be retry-safe instead.
- **Don't `DROP COLUMN` an old column in the same release that stops reading it.** The two pods running mid-rolling-update will have one version that still reads it. Pattern: stop reading (release N), drop column (release N+1).
- **Don't issue a `RENAME TABLE` on a hot table without a maintenance window.** It's atomic but takes a brief metadata lock; for big tables on busy systems that lock window is non-zero.

### B.5 Diagnostics and the `sys` schema

**DO**

- **`EXPLAIN ANALYZE`** (MySQL 8.0.18+) for every new hot-path query. Tree format only. Compare `actual time` against `cost` estimates; if they diverge by >10Ă—, run `ANALYZE TABLE <name>;` to refresh stats and re-plan.
- **`sys.statement_analysis`** for "what's slow right now". Aggregated by digest.
- **`sys.schema_unused_indexes`** before dropping any index.
- **`sys.schema_index_statistics`** for "what indexes are actually doing work".
- **`sys.innodb_lock_waits`** when something feels stalled.
- **`sys.schema_tables_with_full_table_scans`** as a regression alarm after schema changes.
- **`performance_schema`** for deeper instrumentation. Setup tables (`performance_schema.setup_*`) control what gets collected; defaults are fine in dev, may need trimming in prod.

**DON'T**

- **Don't run `EXPLAIN ANALYZE` against prod for an expensive query** â€” it actually executes. Use a staging replica with prod-shape data. (`EXPLAIN` without `ANALYZE` is fine on prod; it doesn't run the query.)
- **Don't enable every Performance Schema instrument in prod.** Some are expensive; the defaults are tuned for general use. Enable extras only with a measurement-driven reason and a plan to disable.

### B.6 Index lifecycle (invisible indexes)

**DO**

- **Stage every index drop via `ALTER TABLE ... ALTER INDEX <n> INVISIBLE` first.** Wait one full week of representative production traffic. If `sys.statement_analysis` and the slow-query log show no regressions, drop. If anything regresses, flip back to `VISIBLE` â€” instant, no rebuild.
- **Stage every new candidate index by adding it `INVISIBLE`** and confirming writes are healthy before flipping to `VISIBLE`.
- **One revision per visibility flip** (and a separate revision for the eventual `DROP`). Single rollback point per change.

**DON'T**

- **Don't drop an index that's been deployed without going through the invisible-index dance.** "I'm sure it's unused" is famous last words; `sys.schema_unused_indexes` only shows what's been read since the server last started.
- **Don't toggle visibility from a psql/mysql REPL.** Through Alembic + `helm upgrade`, always.

### B.7 Configuration changes (`SET PERSIST`)

**DO**

- **`SET PERSIST <var> = <value>`** for live tuning of dynamic variables (e.g. `max_connections`, `slow_query_log`). Writes to `mysqld-auto.cnf` with `SET_USER` / `SET_TIME` audit metadata; survives restart.
- **`SET PERSIST_ONLY <var> = <value>`** for read-only variables that need a restart. Writes-but-defers.
- **Mirror every persisted change back into the Helm values file (`rebuild/infra/k8s/values-prod.yaml`) in the next deploy.** `mysqld-auto.cnf` must not drift from version control. (See [`m5-hardening.md` Â§ Configuration changes](m5-hardening.md).)
- **Inspect via `performance_schema.variables_info WHERE VARIABLE_SOURCE = 'PERSISTED'`** to audit what's been changed when.
- **`RESET PERSIST <var>;`** to remove a persisted setting. The running value is unchanged; use `SET GLOBAL` to revert.

**DON'T**

- **Don't use `SET PERSIST` for replication-topology, binlog-format, or data-dictionary variables** without escalating to the platform team. Those changes have whole-fleet consequences.
- **Don't tune in prod without baseline numbers.** "I bumped `innodb_buffer_pool_size`" with no before/after isn't tuning.

### B.8 Backups, replication, and HA boundaries

**DO**

- **Treat backups as a platform-team responsibility** (per `rebuild.md` Â§9). The rebuild's job is to be backup-friendly: idempotent migrations, no schema drift, ROW-format binlog assumed for PITR.
- **Know which queries can run on a read replica** (if/when one exists). Reads of `chat.history` for a `/s/:token` snapshot can; the SSE streaming write path can't.

**DON'T**

- **Don't deploy InnoDB Cluster, Group Replication, or MySQL Router** as part of the rebuild. Single managed instance only. (`rebuild.md` Â§9.)
- **Don't issue a backup-style query (`mysqldump`, `xtrabackup`, full-table read) from the application user.** That's a platform-team scoped role.

## C. Features we have explicitly declined (and why)

When something below comes up in code review, point at this section. Each item carries the rationale so you don't have to re-derive it.

| Feature | Status | Reason | Where to revisit |
|---|---|---|---|
| `BINARY(16)` UUID PKs | Declined | UUIDv7 strings already give insertion locality; the 55% storage win doesn't justify the SQLAlchemy `TypeDecorator` complexity. | `rebuild.md` Â§9; `MYSQL_FEATURE_AUDIT.md` Â§2 |
| InnoDB Cluster / Group Replication / MySQL Router | Declined | Single managed MySQL is sufficient for internal scale; HA is platform-team-owned. | `rebuild.md` Â§9 |
| Object store (S3) for files | Declined | `MEDIUMBLOB` + 5 MiB cap suffices; `FileStore` interface lets us swap if we ever outgrow it. | `rebuild.md` Â§9 |
| `FULLTEXT` + `MATCH AGAINST` (with or without `ngram` parser) | Out of scope for M1+ | `LIKE %q%` + `JSON_SEARCH` is good enough at our scale; benchmarks show `ngram` actually loses to `LIKE` on chat-style queries. | M5 if `?q=` becomes a complaint vector |
| Multi-valued indexes (`MEMBER OF`, `JSON_CONTAINS`) | N/A | Reactions are in their own row table, not collapsed into JSON. The MVI question doesn't arise. | If ever schema collapses reactions into JSON (don't) |
| `JSON_TABLE` | Declined | Chat history is a tree, not a relational shape; flattening defeats the design. | Never |
| Window functions / hash-join hints / `LATERAL` derived tables | Not used | No current workload needs them; trust the optimiser. The 8.0 planner picks hash join automatically. | If a real query plan demonstrates a need |
| Histograms (`ANALYZE TABLE ... UPDATE HISTOGRAM`) | Not used | All hot lookups combine high-cardinality `user_id` with a composite-indexed column. | If `EXPLAIN ANALYZE` shows estimate skew on a low-cardinality column |
| MariaDB compatibility | Not supported | We are MySQL 8.0 only; some helpers (`STORED` generated column syntax, multi-valued indexes if added later) are MySQL-specific. | Never |
| Stored procedures, triggers, views beyond `sys.*` | Not used | Application logic lives in the app, not the DB. Easier to test, version, and observe. | Never |
| Multi-statement query execution from app | Disabled by driver default | Avoid; one statement per `execute()`. | Never |
| Sharding / horizontal partitioning | Out of scope | Not at the scale that requires it. Revisit if `chat` exceeds tens of millions of rows. | Far future |
| Session variables for application state | Don't | Use Redis or app-side state; session vars make queries non-portable across pods and complicate connection pooling. | Never |

## D. Pre-PR checklist for agents

Before opening a PR that touches the database (schema, query, or DAO), run through this list. If you can't truthfully tick a box, fix it or call it out in the PR description.

**Schema changes**

- [ ] Migration uses M0 helpers (`*_if_not_exists` / `*_if_exists`), no bare `op.*`.
- [ ] `ADD COLUMN` uses `ALGORITHM=INSTANT, LOCK=DEFAULT` (the helper default). If not, justified in a comment.
- [ ] Every constraint and index has an explicit name (`fk_*`, `uq_*`, `ix_*`, `chk_*`).
- [ ] Every FK has a deliberate `ON DELETE` choice (CASCADE, SET NULL, RESTRICT) documented in the model docstring.
- [ ] `BIGINT` epoch ms for any timestamp; no new `DATETIME`/`TIMESTAMP` columns without a justification.
- [ ] PK is UUIDv7 via `default=new_id` (or `default=app.core.ids.new_id`); no `uuid.uuid4`.
- [ ] `mysql_engine`, `mysql_charset`, `mysql_collate` set (or inherited via the helper).
- [ ] `upgrade()` and `downgrade()` are both idempotent (`test_upgrade_head_is_idempotent` / `test_downgrade_base_is_idempotent` will catch you if not).
- [ ] Schema and data backfill are in **separate** revisions if both are needed.

**New query / endpoint**

- [ ] `EXPLAIN ANALYZE` run against staging with prod-shape data; query plan attached to PR description if it touches a table over 10k rows.
- [ ] `LIMIT` is set; pagination is cursor-based on `(sort_column, id)`, not `OFFSET`.
- [ ] No `SELECT *`; columns are explicit.
- [ ] `SELECT ... FOR UPDATE` only with `SKIP LOCKED` or `NOWAIT`.
- [ ] Indexes that the query relies on exist; if a new index is needed, added in this PR or a parent PR.
- [ ] Transaction scope is the smallest that preserves the invariant; no external network calls inside `BEGIN`.
- [ ] Per-statement timeout (`MAX_EXECUTION_TIME(<ms>)`) on anything with an SLO.
- [ ] Parameterised; no string interpolation of user input.
- [ ] No N+1 (use `JOIN`, CTE, or batched `IN (...)`).

**New ORM model**

- [ ] Lives under `app/models/<table>.py` (one file per table).
- [ ] PK column type matches every other PK (`String(36)` UUIDv7 string, `default=new_id`). Never `CHAR(36)` â€” see Â§B.2 / Â§I.
- [ ] `__table_args__` declares indexes with explicit names matching the migration.
- [ ] Module-level `from __future__ import annotations` and absolute imports (`from app.db.base import Base`).
- [ ] Re-exported from `app/models/__init__.py`.

**Index changes**

- [ ] Drop = `INVISIBLE` first (one revision), wait one week, then `DROP` (second revision).
- [ ] Add = `INVISIBLE` first (one revision), confirm write health, then `VISIBLE` (second revision).
- [ ] `sys.schema_unused_indexes` and `sys.schema_index_statistics` consulted before proposing a drop.

**Production tuning**

- [ ] `SET PERSIST` change has a follow-up PR mirroring it into `values-prod.yaml`.
- [ ] Variable is on the runbook's allow-list (or the platform team has been consulted).
- [ ] Before/after numbers attached to the change.
